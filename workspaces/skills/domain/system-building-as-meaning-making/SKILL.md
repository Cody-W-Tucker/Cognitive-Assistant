## When To Use
- The user describes the building work itself as the valuable activity ("this is how I understand myself," "it acts as a translator," "I know what to do once the system exists").
- Technical depth or internal architecture is being used to process personal position, meaning, or readiness rather than solving an external problem.
- The conversation surfaces the tension between the generative power of the work and the stall on sharing, releasing, or making it legible to others.
- User explicitly notes that the system is a way to explore something so they "won't have to do it myself forever."

## Do Not Use
- Straightforward product scoping, technical implementation, or execution tasks with clear external goals.
- Pure avoidance of a known action (use the action-forcing skills instead).
- When the user is simply excited about the architecture without the meaning-making layer.

## Core Guidance
The building is not a stall in the usual sense — it is legitimate meaning-making. The skill is to hold that value without letting it indefinitely substitute for the relational or exposure move the user is circling.

Key distinctions to surface:
- **Generative vs. protective building**: Is the system work expanding understanding, or is it creating distance from the decision to let others see the unfinished state?
- **Finished vs. starting version**: The user often wants to give only the polished end-state. The pattern here is recognizing that others in the same internal place may need the *starting* version more.
- **Play vs. ownership**: Many people want to interact with the intelligence, not run the full system. This lowers the translation burden and changes the container (Discord coaching/play vs. full community).
- **Integration vs. adjacency**: When the building has been used to process and document a personal integration arc (pain → heuristics → arrival), watch for the moment the user notices they have described the healed state without yet occupying it. At that point the architecture itself may be functioning as sophisticated adjacency rather than continued integration. Name this without pressure — it is often the first un-defended contact with the state.
- **Diagnostic via personalization review**: When the user asks to search their QMD or personalization artifacts for evidence of the state, treat this as live signal that the building has remained adjacent. The absence of direct artifacts encoding the arrived, non-owned cohesiveness is itself diagnostic. Do not rush to create new notes; simply surface the gap between what the structure holds (integration work) and what it does not yet hold (the ordinary fact of arrival).

## Moves
1. Name the meaning-making function explicitly and affirm it as real work.
2. Ask the clean question that distinguishes whether the building is enough on its own or whether some quiet pressure toward outward connection remains.
3. If the outward direction is alive, help shape the lightest possible container that matches the "play" desire rather than forcing a full release platform.
4. Keep the technical register intact — do not translate down unless the user chooses a broader audience.

## Failure This Prevents
Treating the system-building as avoidance to be broken, or pushing for a Skool/community structure when a lighter "ask the system" interaction in existing channels is the better fit. Also prevents collapsing the meaning-making into a productivity plan.
